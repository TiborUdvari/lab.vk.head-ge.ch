06 - Connect the dots

Children’s connect-the-dots puzzles as a vehicle for D3’s path generator. Each shape is an array of {x, y} points loaded from JSON; d3.line() turns them into an SVG &lt;path&gt; string.

Source: dataset from Williams College CS 326 — Connect The Dots lab. Original zip here. Files copied into /assets/connect-the-dots/.

Loading and normalizing

Each puzzle’s coordinates use a different range (leaf is ~600 px wide, star ~270). We normalize each shape to fit in a 300 × 300 box, centered, with aspect ratio preserved — so the SVG can render any shape at the same scale.

function normalize(points, size = 300) {
  const xExtent = d3.extent(points, d => d.x);
  const yExtent = d3.extent(points, d => d.y);
  const range = Math.max(xExtent[1] - xExtent[0], yExtent[1] - yExtent[0]);
  const scale = size / range;
  const cx = (xExtent[0] + xExtent[1]) / 2;
  const cy = (yExtent[0] + yExtent[1]) / 2;
  return points.map(p => ({
    x: (p.x - cx) * scale + size / 2,
    y: (p.y - cy) * scale + size / 2
  }));
}
const shapes = {
  star:   normalize(await d3.json("data/connect-the-dots/star.json")),
  leaf:   normalize(await d3.json("data/connect-the-dots/leaf.json")),
  tree:   normalize(await d3.json("data/connect-the-dots/tree.json")),
  runner: normalize(await d3.json("data/connect-the-dots/runner.json")),
  crown:  normalize(await d3.json("data/connect-the-dots/crown.json"))
};

The line generator + curve interpolation

d3.line() connects points; the curve decides how — straight segments, smooth splines, etc. Using the *Closed variants makes the curve wrap from the last point back to the first, no "Z" needed.

  • linear — straight segments, the literal connect-the-dots
  • catmullRom — passes through every point with a smooth cubic interpolation
  • cardinal — similar but uses tangents (tension 0.5 by default)
  • basis — B-spline; the curve does not pass through control points, it’s pulled toward them — gives a more “blobby” silhouette

All four are closed curves here, so the line wraps the dots into a silhouette automatically.

const curves = {
  linear:     d3.curveLinearClosed,
  catmullRom: d3.curveCatmullRomClosed,
  cardinal:   d3.curveCardinalClosed,
  basis:      d3.curveBasisClosed
};

const lineGen = d3.line()
  .x(d => d.x)
  .y(d => d.y)
  .curve(curves[curveType]);

Path with self-drawing animation

The classic D3 “drawing in” trick:

  1. Set the path d, then measure its total length with node().getTotalLength().
  2. Set stroke-dasharray and stroke-dashoffset to that length — the entire stroke is now “dashed off-screen” (the dash gap covers the whole line).
  3. Transition stroke-dashoffset to 0 — the dash slides into place, looking like the line is being drawn.
  4. Fade in the fill afterward (otherwise the silhouette pops in fully colored from the start).

The cell depends on shapeName, lineGen (which depends on curveType), and redrawClick — any of those changing replays the animation.

{
  const shape = shapes[shapeName];
  redrawClick;   // dependency declaration — clicking Redraw re-runs the cell

  const pathSel = g.selectAll("path.shape")
    .data([null])
    .join("path")
      .attr("class", "shape")
      .style("fill", "tomato")
      .style("stroke", "tomato")
      .style("stroke-width", 1.5);

  // 1. set the path data using the current line generator
  pathSel.attr("d", lineGen(shape));

  // 2. measure the resulting path's length in pixels
  const length = pathSel.node().getTotalLength();

  // 3. set up the dash trick: a single dash equal to the full length,
  //    offset by the same amount → the line is "drawn off-screen" initially
  pathSel
    .attr("stroke-dasharray", length)
    .attr("stroke-dashoffset", length)
    .style("fill-opacity", 0)
    .style("opacity", showPath ? 1 : 0);

  // 4. animate offset to 0 → line draws in. Then fade in the fill.
  pathSel.transition().duration(2000).ease(d3.easeQuadInOut)
    .attr("stroke-dashoffset", 0)
    .transition().duration(500)
    .style("fill-opacity", showPath ? 0.15 : 0);
}

Dots and numbers — keyed by index

Each point gets a circle. Numbers are optional — useful for sparse puzzles like star/leaf, noisy for the 90-point ones.

{
  const shape = shapes[shapeName];
  const dotR = shape.length < 30 ? 8 : 3;

  // dots
  g.selectAll("circle.dot")
    .data(shape, (d, i) => i)
    .join(
      enter => enter.append("circle")
        .attr("class", "dot")
        .style("fill", "white")
        .style("stroke", "black")
        .style("stroke-width", 1)
        .attr("r", 0)
        .attr("cx", d => d.x)
        .attr("cy", d => d.y),
      update => update,
      exit => exit.transition().duration(400).attr("r", 0).remove()
    )
    .transition().duration(750)
      .attr("cx", d => d.x)
      .attr("cy", d => d.y)
      .attr("r", dotR);

  // numbers
  g.selectAll("text.number")
    .data(shape, (d, i) => i)
    .join(
      enter => enter.append("text")
        .attr("class", "number")
        .attr("text-anchor", "middle")
        .attr("dominant-baseline", "middle")
        .style("font-family", "sans-serif")
        .style("fill", "black")
        .style("opacity", 0)
        .attr("x", d => d.x)
        .attr("y", d => d.y),
      update => update,
      exit => exit.transition().duration(400).style("opacity", 0).remove()
    )
    .text((d, i) => i + 1)
    .attr("font-size", shape.length < 30 ? 9 : 7)
    .transition().duration(750)
      .attr("x", d => d.x + (shape.length < 30 ? 0 : 8))
      .attr("y", d => d.y + (shape.length < 30 ? 0 : -8))
      .style("opacity", showNumbers ? 1 : 0);
}

How the pieces fit

  • Cell 4 defines normalize — fits any point array into a centered, aspect-preserved 300 × 300 box.
  • Cell 5 loads all five JSON files in parallel via await d3.json(...). Top-level await is supported in module cells; the runtime waits for the cell’s promise before letting dependents run. Each shape gets normalized once at load time.
  • Cells 7 / 9 / 10 / 20 / 21 are the controls — shape, show path, show numbers, curve type, redraw button. Cell 8 displays the SVG amidst them.
  • Cell 14 defines lineGen, depending on curveType. Switching curves rebuilds the generator and the path cell re-runs.
  • Cell 16 does both the data-to-path conversion and the self-drawing animation. The stroke-dasharray / stroke-dashoffset trick converts a static path into an animated reveal — works on any path of any complexity, you only need its total length.
  • Cell 18 renders dots and numbers, with sizes that adapt to point count.

On getTotalLength()

Every SVG path element exposes .getTotalLength() — the path’s stroked length in user-space units. Combined with .getPointAtLength(t), you can do a lot of path-following effects (markers tracing a line, text along a curve, particles released along a contour). Worth bookmarking.

On curve choice

For connect-the-dots specifically:

  • linear is the literal puzzle — what kids draw with a pencil.
  • catmullRom smooths corners while still passing through every point — usually the most natural-looking option for organic shapes (leaf, runner).
  • cardinal is similar but uses tangents at each point; subtly different curvature.
  • basis doesn’t pass through the points — they act as control points pulling the curve, which makes the silhouette more abstract.

Try the same shape with all four to see the same data interpreted four ways. That’s the lesson: the points are data, the curve is presentation.

Where to take it next

  • Curve along an SVG path: path.getPointAtLength(t) lets you place markers, characters, or sub-shapes at fractional positions along the line. Animate t from 0 to length, attach a circle, you have a tracer.
  • d3.drag() on the dots: mutate the underlying array, watch the path follow. A shape editor in 20 lines.
  • More puzzles: drop more JSON files in /assets/connect-the-dots/ matching [{x, y}, ...] and add them to cell 5.
